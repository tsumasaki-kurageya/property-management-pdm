# 業務エクスプローラー 品質検証

## 目的

業務エクスプローラーが、画面幅、入力方法、GitHub Pagesの公開環境に依存せず、初学者でも主要操作を完了できることを継続的に確認する。

## 自動検証

Playwrightで次の3画面幅を検証する。

| 区分 | 画面幅 | 主な確認内容 |
|---|---:|---|
| PC | 1440 × 900 | 3ペイン、表示切替、検索、URL履歴、詳細表示 |
| タブレット | 900 × 1100 | 2列化、詳細ペインの下段表示、横はみ出し |
| スマートフォン | 390 × 844 | 業務選択ドロワー、図・説明・全体位置の切替、タッチ操作 |

### 代表シナリオ

`BM-09-06 点検異常を判定する`を起点として、少なくとも次を確認する。

- この前の仕事: 点検値を記録する
- 通常の次の仕事: 点検結果を承認する
- 条件で分かれる仕事: 不具合を受け付ける
- 図と同じ内容をテキスト一覧でも確認できる
- 局所表示のノード数が過剰にならない

### キーボード

- 表示方法タブを左右矢印、Home、Endで切り替えられる
- 検索欄から矢印キーで結果へ移動し、Enterで選択できる
- 選択後に中央マップ、右詳細、ライフサイクルが同期する
- フォーカス位置を視認できる

### URLと履歴

- 選択業務、表示方法、関係分類がURLへ反映される
- 戻る、進む、再読み込みで同じ状態を復元できる
- クエリ付きURLへ直接アクセスできる
- 旧`node`パラメーターと不正値を安全に補正できる

### アクセシビリティ

`@axe-core/playwright`を利用し、エクスプローラー領域に重大度`critical`または`serious`のWCAG 2.0／2.1 A・AA違反がないことを確認する。

自動検査だけでは操作の理解しやすさや読み上げ順序の妥当性を完全には判定できないため、下記の手動確認も残す。

## 手動確認

- 専門語を知らなくても「仕事の流れ」「全体から見る」「関係から探す」の違いを推測できる
- 色を見分けられなくても、ラベルと凡例から関係を識別できる
- スクリーンリーダーで見出し、タブ、ツリー、折りたたみ、状態通知を順に把握できる
- 200%拡大時にも主要操作が画面外へ失われない
- スマートフォンでページスクロールと図のドラッグが競合しない
- ライトテーマとダークテーマの両方でフォーカスを視認できる

## 実行方法

初回のみChromiumをインストールする。

```bash
npx playwright install chromium
```

本番ビルドを生成してからE2Eテストを実行する。

```bash
npm run build
npm run test:e2e
```

失敗時のHTMLレポートを開く。

```bash
npm run test:e2e:report
```

## GitHub Actions

- `Docs CI`: 本番ビルド後、PC・タブレット・スマートフォンのE2Eを実行する
- `Deploy GitHub Pages`: デプロイ後、公開URLに対して共有URLの直接アクセスと再読み込みを検証する
- E2E失敗時はPlaywrightのレポート、スクリーンショット、トレースをartifactとして保存する

## 合格基準

- `npm run verify`が成功する
- 全Playwrightプロジェクトが成功する
- 公開後スモークテストが成功する
- 代表シナリオの3関係が登録データから表示される
- 重大な自動検出可能アクセシビリティ違反が0件である
- ドキュメント全体に横スクロールが発生しない
