# 業務エクスプローラー 品質検証

## 目的

業務エクスプローラーが、画面幅、入力方法、GitHub Pagesの公開環境に依存せず、18業務領域から個別業務を選び、その業務を含む横断プロセスと詳細へ到達できることを継続的に確認する。

## 自動検証

Playwrightで次の3画面幅を検証する。

| 区分 | 画面幅 | 主な確認内容 |
|---|---:|---|
| PC | 1440 × 900 | 全体地図、プロセス図、右詳細、検索、URL履歴 |
| タブレット | 900 × 1100 | 領域展開、プロセス図、詳細ペイン、横はみ出し |
| スマートフォン | 390 × 844 | タッチによる領域展開、プロセス・詳細切替 |

### 主要シナリオ

- 初期画面には18業務領域だけを表示し、詳細やプロセス図を常設しない
- 業務領域のホバー、フォーカス、タップで所属業務を全件表示する
- 業務ノードの選択で、その業務を含むプロセスを開始から終了まで表示する
- 複数プロセスを正本順で表示し、選択業務を維持したまま切り替える
- プロセス内、右詳細、検索の業務選択で共通の状態遷移を実行する
- 業務詳細で開始契機、完了条件、入力、成果物、担当者、法令、手順、帳票、関連業務を確認する
- 旧3表示タブ、旧領域ツリー、下部ライフサイクルを表示しない

### 入力方法

- マウスのホバーとクリックで主要フローを完了できる
- キーボードのフォーカス、矢印キー、Enter、Escapeで領域展開、検索、選択を完了できる
- タッチで領域を展開し、業務、プロセス、詳細を選択できる
- フォーカス位置と選択状態を色以外のラベルでも確認できる

### URLと履歴

- URLへ保存する状態は選択業務IDと表示プロセスIDに限定する
- 戻る、進む、再読み込みで同じ状態を復元する
- GitHub Pagesのベースパスを含む共有URLへ直接アクセスできる
- 旧`node`、`item`、`view`、`type`パラメーターと不正値を安全に正規化する

### アクセシビリティ

`@axe-core/playwright`を利用し、エクスプローラー領域に重大度`critical`または`serious`のWCAG 2.0／2.1 A・AA違反がないことを確認する。

## 手動確認

- 18業務領域ノードと選択可能な個別業務ノードの違いを理解できる
- 矢印を「次に進む」と理解でき、条件分岐を日本語ラベルで判別できる
- 200%拡大時にも主要操作が画面外へ失われない
- スマートフォンでページスクロールと図の操作が競合しない
- ライトテーマとダークテーマの両方でフォーカスと選択状態を視認できる
- GitHub Pages上で表示、検索、業務選択、プロセス切り替え、詳細リンクが動作する

## 実行方法

初回のみChromiumをインストールする。

```bash
npx playwright install chromium
```

本番ビルドを生成してからE2Eテストを実行する。

```bash
npm run verify
npm run test:e2e
```

失敗時のHTMLレポートを開く。

```bash
npm run test:e2e:report
```

## GitHub Actions

- `Docs CI`: 正本参照検証、本番ビルド、PC・タブレット・スマートフォンのE2Eを実行する
- `Deploy GitHub Pages`: デプロイ後、公開URLに対して共有URLの直接アクセスと再読み込みを検証する
- E2E失敗時はPlaywrightのレポート、スクリーンショット、トレースをartifactとして保存する

## 合格基準

- `npm run verify`が成功する
- 全Playwrightプロジェクトが成功する
- 公開後スモークテストが成功する
- 重大な自動検出可能アクセシビリティ違反が0件である
- PC・タブレット・スマートフォンで主要情報へ到達できる
- ドキュメント全体に横スクロールが発生しない
