# 業務エクスプローラー再設計仕様

## 1. 目的

業務エクスプローラーは、ビルメンテナンス業務全体を俯瞰し、関心のある業務を選択して、その業務が含まれる開始から終了までの業務プロセスと詳細情報を確認するための画面とする。

利用者が理解する順序は次のとおりとする。

1. ビルメンテナンス業務全体はどのように進むか
2. 関心のある業務はどこにあるか
3. その業務は、開始から終了までどのようにつながるか
4. その業務には、どの入力・成果物・担当者・法令・手順が関係するか

業務カタログの階層、局所的な前後関係、帳票・担当者・法令との関連、ライフサイクルを同格に並べるグラフ閲覧画面にはしない。

---

## 2. 機能と優先順位

中核機能は次の4つとする。

1. 全体の業務地図
2. 選択業務が含まれる業務プロセス
3. 業務の詳細
4. 検索

4機能を同列のタブにはしない。全体から個別へ掘り下げる構造とする。

```text
全体の業務地図
    ↓ 代表業務を選択
業務プロセス全体
    ＋
選択業務の詳細
```

検索は独立画面ではなく、全体地図と業務プロセスの両方から利用できる共通入口とする。

URL共有、再読み込み、ブラウザの戻る・進むによる状態復元は補助機能とする。中核機能の設計や実装を複雑にしてまで優先しない。

---

## 3. 画面状態

### 3.1 全体地図状態

初期表示状態。

中央には、ビルメンテナンス業務全体の進行段階を表す大分類ノードを表示する。

大分類は次の7段階を基本とする。

1. 要求・契約
2. 立ち上げ
3. 計画・準備
4. 実施・受付
5. 確認・報告
6. 復旧・精算
7. 改善・更新・終了

### 3.2 代表業務プレビュー状態

大分類ノードへマウスオーバーまたはキーボードフォーカスした状態。

大分類ノードの周囲に、その段階を理解するための代表業務ノードを一時表示する。

この状態では業務選択を確定しない。

### 3.3 業務プロセス状態

代表業務、検索結果、業務プロセス内の別業務、詳細内の関連業務から業務を選択した状態。

中央表示を全体地図から、選択業務が含まれる業務プロセス全体へ置き換える。

全体地図、大分類ノード、代表業務ノードは中央表示から消す。全体地図と業務プロセスを同時表示しない。

右側には選択業務の詳細を表示する。

### 3.4 状態遷移

```text
全体地図
  ├─ 大分類へホバー／フォーカス／タップ
  │     → 代表業務をプレビュー
  │
  ├─ 代表業務をクリック／Enter
  │     → 業務プロセス
  │
  └─ 検索結果の業務を選択
        → 業務プロセス

業務プロセス
  ├─ プロセス内の別業務を選択
  │     → 選択業務を変更
  │
  ├─ 別シナリオを選択
  │     → 表示プロセスを変更
  │
  ├─ 検索結果を選択
  │     → 対応する業務またはプロセスへ移動
  │
  └─ 全体の業務地図へ戻る
        → 全体地図
```

## 4. 全体の業務地図

### 4.1 表現対象

全体地図は、個別業務の詳細な順序ではなく、ビルメンテナンス業務全体の進行段階を表す。

```text
要求・契約
  ↓
立ち上げ
  ↓
計画・準備
  ↓
実施・受付
  ↓
確認・報告
  ↓
復旧・精算
  ↓
改善・更新・終了
```

初見の利用者が「何の全体図か」を理解できるタイトルと説明を表示する。

### 4.2 大分類ノード

大分類ノードは業務選択対象ではなく、クリック不可とする。

役割は次の2つだけとする。

- 業務全体の段階を示す
- 代表業務を表示する起点になる

操作は次のとおりとする。

- マウスオーバーで代表業務を一時表示する
- キーボードフォーカスで代表業務を表示する
- タッチ端末ではタップで代表業務を展開する
- 大分類ノード自体をEnterやクリックで選択しない

### 4.3 代表業務ノード

代表業務ノードは、全体地図から個別業務へ入る主な起点とする。

代表業務ノードのみクリックまたはEnterで選択できる。

代表業務を選択すると、当該業務を選択業務として確定し、その業務を含む代表的な業務プロセスを表示する。

代表業務は、各段階の意味を理解するために必要な業務を選ぶ。件数の均等配分は目的にしない。

選定では次を考慮する。

- 段階の開始または完了を表す
- 情報・状態・責任の引渡し点である
- 重要判断または例外分岐を含む
- 複数領域を横断する
- 初学者が段階の意味を理解しやすい

---

## 5. 業務プロセス

### 5.1 定義

業務ノードは一つの仕事を表す。

業務プロセスは、複数の業務ノードを開始から終了まで意味のある順序でつないだ業務シナリオを表す。

一つの業務が複数の業務プロセスへ含まれることを許容する。

### 5.2 表示範囲

選択業務の前後1ホップだけを表示しない。

選択業務が含まれる業務シナリオを開始から終了まで表示する。

```text
開始
  ↓
点検を計画する
  ↓
点検を実施する
  ↓
点検値を記録する
  ↓
点検異常を判定する  ← 選択中
  ├─ 正常の場合 → 点検結果を承認する → 報告する → 完了
  └─ 異常の場合 → 不具合を受け付ける → 一次対応 → 修繕・復旧の流れへ
```

横方向を時間・進行方向とする。条件分岐は上下へ展開する。

選択業務は「選択中」という文字と枠・背景で強調し、色だけに依存しない。

### 5.3 複数プロセスへの所属

一つの業務が複数プロセスへ含まれる場合は、代表プロセスを初期表示する。

画面上部から別プロセスへ切り替えられるようにする。

代表プロセスは正本データで明示し、UIが接続数などから推測しない。

---

## 6. 矢印と経路の表現

矢印は1種類だけとする。

```text
矢印 ＝ 次に進む
```

利用者へ、線の色、太さ、破線、矢印形状の意味を覚えさせない。

特殊な意味は線種ではなく日本語で表す。

- 条件分岐：`正常の場合`、`異常の場合`
- 別プロセスへの引渡し：`修繕・復旧の流れへ`
- 差戻し・再実施：`点検の再実施へ`
- 開始・終了：`開始`、`完了`

合流専用の線種や記号は設けない。複数経路が同じ業務へ進む場合は、同じ業務ノードへ接続する。

---

## 7. 業務の詳細

選択業務の詳細は、業務プロセスとは別の右ペインに表示する。

表示候補は次のとおりとする。

- 業務ID・業務名
- 概要
- 開始契機
- 完了条件
- 入力
- 成果物
- 実施者
- 判断者・承認者
- 法令・基準
- 作業手順
- チェックリスト・帳票
- 関連業務
- 参照元

帳票、担当者、法令、類似業務などを中央の業務プロセスへノードとして混在させない。

中央は業務の進行、右ペインは業務に付随する情報という役割を維持する。

---

## 8. 検索

検索は全体地図と業務プロセスの両方から利用できる共通機能とする。

検索対象候補は次のとおりとする。

- 業務ID
- 業務名
- 業務説明
- 業務領域
- ライフサイクル段階
- 業務プロセス名
- 入力
- 成果物
- 担当者
- 法令・基準
- 作業手順
- チェックリスト・帳票

業務を選択した場合は、その業務を含む代表プロセスと業務詳細を表示する。

業務プロセスを選択した場合は、当該プロセスを開始から終了まで表示する。

---

## 9. 業務プロセスの正本データ

業務プロセスは、既存の接続関係を無制限に探索して自動生成しない。

各プロセスについて、少なくとも次を正本データとして定義する。

- プロセスID
- プロセス名
- 初学者向け説明
- 代表プロセスかどうか
- 開始点
- 終了点
- 業務ノード
- 次工程
- 条件分岐ラベル
- 別プロセスへの引渡し先
- 同一プロセス内の再実施先
- 当該プロセスに含まれる業務ID一覧

概念例は次のとおりとする。

```yaml
id: inspection-abnormality
name: 点検計画から異常対応・報告まで
representative: true
steps:
  - id: plan
    businessId: BM-09-01
    next: execute
  - id: execute
    businessId: BM-09-04
    next: judge
  - id: judge
    businessId: BM-09-06
    branches:
      - label: 正常の場合
        next: approve
      - label: 異常の場合
        next: restoration-handoff
  - id: restoration-handoff
    handoffProcessId: restoration
    label: 修繕・復旧の流れへ
```

## 10. 既存実装の扱い

既存実装の再利用を前提条件にしない。

新しい情報構造と操作を単純に実現できる場合に限り、データ処理や小さなUI部品を再利用する。

既存の状態管理、画面構造、コンポーネント境界、採用技術が再設計の妨げになる場合は、削除または作り直す。

既存コードを残すことより、次を優先する。

- 利用者が画面の意味を理解できること
- 画面状態と操作が単純であること
- 正本データとUIの責任が明確であること
- 不要な互換性維持をしないこと

現在の1ホップフロー、`全体から見る`、`関係から探す`、3表示タブ、下部ライフサイクルは、新設計の前提にはしない。

React Flow、ELK、現在の検索、詳細パネル、URL・履歴同期も必須技術・必須部品とはしない。実装時に適合する部分だけを採用する。

---

## 11. URL・ブラウザ履歴の位置づけ

URL共有、再読み込み、ブラウザの戻る・進むによる状態復元は補助機能とする。

中核機能が完成し、共有や状態復元の必要性が確認できた段階で対応範囲を判断する。

対応する場合も、選択業務IDと表示中の業務プロセスID程度に限定する。代表業務のホバー・フォーカス状態は保持しない。

URL・履歴対応のために画面状態やコンポーネント構造を複雑化させない。

---

## 12. 業務エクスプローラー再設計で採用しない設計

本仕様の対象である業務エクスプローラーでは、次の設計を採用しない。

- 大分類ノードをクリックして業務選択する
- 全体地図と個別業務プロセスを同時に表示する
- 選択業務の前後1ホップだけを業務プロセスと呼ぶ
- 帳票、担当者、法令を中央の業務プロセス図へ混ぜる
- 線の色、太さ、破線で意味を区別する
- ホバーだけで選択業務を確定する
- 接続関係を無制限探索してエンドツーエンドプロセスを自動推測する
- 4機能を同列タブとして並べる
- 既存実装を残すことを優先して新しい情報構造を歪める

---

## 13. 未確定事項

実装前に次を確定する。

1. 全体地図の大分類ノード配置
2. 各大分類の代表業務数と選定結果
3. 初期作成する業務プロセスの範囲と優先順位
4. 業務プロセス正本の保存形式
5. 分岐が多いプロセスの段階表示・折りたたみ方法
6. 全体地図へ戻った際に、直前の大分類プレビューを復元するか
7. タッチ端末で代表業務群を閉じる操作
8. 検索対象を初期版でどこまで広げるか

未確定事項は推測で実装せず、個別に合意してから確定する。

---

## 14. 受け入れ基準

- 初期画面を見て、ビルメンテナンス業務全体の地図であると理解できる
- 大分類ノードが業務選択対象ではないことが分かる
- 大分類へのホバー・フォーカス・タップで代表業務を確認できる
- 代表業務のクリック・Enterから業務プロセスへ移動できる
- 全体地図と業務プロセスが同時表示されない
- 選択業務が含まれる流れを開始から終了まで確認できる
- 一つの業務が含まれる複数プロセスを切り替えられる
- 矢印の意味を「次に進む」だけで理解できる
- 条件分岐、別プロセスへの引渡し、再実施を日本語で理解できる
- 業務詳細で入力、成果物、担当者、法令、手順を確認できる
- 全体地図と業務プロセスの両方から検索を利用できる
- マウス、キーボード、タッチで主要操作を完了できる
