# 業務エクスプローラー再設計仕様

## 1. 目的

業務エクスプローラーは、ビルメンテナンス業務全体を俯瞰し、関心のある業務を選択して、その業務が含まれる開始から終了までの業務プロセスと詳細情報を確認するための画面とする。

現在のように、業務カタログの階層、局所的な前後関係、帳票・担当者・法令との関連、ライフサイクルを同格に並べるグラフ閲覧画面にはしない。

利用者が最初に理解すべき問いは次の順序である。

1. ビルメンテナンス業務全体はどのように進むか
2. 関心のある業務はどこにあるか
3. その業務は、どの業務プロセスの中で、開始から終了までどうつながるか
4. その業務には、どの入力・成果物・担当者・法令・手順が関係するか

---

## 2. 情報アーキテクチャ

業務エクスプローラーは、次の4要素で構成する。

1. 全体の業務地図
2. 選択業務が含まれる業務プロセス
3. 業務の詳細
4. 検索

4要素を同列のタブにはしない。全体から個別へ掘り下げる階層構造とする。

```text
全体の業務地図
    ↓ 代表業務を選択
業務プロセス全体
    ＋
選択業務の詳細
```

検索は独立画面ではなく、全状態から利用できる共通入口とする。

---

## 3. 画面状態

### 3.1 全体地図状態

初期表示状態。

中央には、ビルメンテナンス業務全体を表す大分類ノードを表示する。

大分類は7つのライフサイクル段階を基本とする。

1. 要求・契約
2. 立ち上げ
3. 計画・準備
4. 実施・受付
5. 確認・報告
6. 復旧・精算
7. 改善・更新・終了

大分類ノードは業務ではなく、業務のまとまり・段階を表す表示要素である。

### 3.2 代表業務プレビュー状態

大分類ノードへマウスオーバーまたはキーボードフォーカスした状態。

大分類ノードの周囲に、当該段階を理解するための代表業務ノードを表示する。

この状態は一時的なプレビューであり、選択業務、URL、ブラウザ履歴を変更しない。

### 3.3 業務プロセス状態

代表業務、検索結果、共有URL、業務プロセス内の別業務、詳細内の関連業務のいずれかから業務を選択した状態。

中央表示を全体地図から、選択業務が含まれる業務プロセス全体へ置き換える。

全体地図、大分類ノード、代表業務ノードは中央表示から消す。

右側には選択業務の詳細を表示する。

### 3.4 状態遷移

```text
全体地図
  ├─ 大分類へホバー／フォーカス
  │     → 代表業務プレビュー
  │
  ├─ 検索結果の業務を選択
  │     → 業務プロセス
  │
  └─ 代表業務をクリック／Enter
        → 業務プロセス

業務プロセス
  ├─ プロセス内の別業務を選択
  │     → 同じプロセス内で選択業務を変更
  │
  ├─ 別シナリオを選択
  │     → 表示する業務プロセスを変更
  │
  ├─ 検索結果を選択
  │     → 対応する業務またはシナリオへ移動
  │
  └─ 全体の業務地図へ戻る
        → 全体地図
```

全体地図と個別業務プロセスを同時表示しない。

---

## 4. 全体の業務地図

### 4.1 表現対象

全体地図は、個別業務の詳細な順序ではなく、ビルメンテナンス業務全体の進行段階を表す。

```text
要求・契約
  ↓
立ち上げ
  ↓
計画・準備
  ↓
実施・受付
  ↓
確認・報告
  ↓
復旧・精算
  ↓
改善・更新・終了
```

初見の利用者が「何の全体図か」を理解できるタイトルと説明を表示する。

### 4.2 大分類ノード

大分類ノードはクリック不可とする。

大分類ノードの役割は次だけに限定する。

- 業務全体の段階を示す
- 代表業務を展開する起点になる

大分類ノードを業務選択の対象にはしない。

#### マウス

- マウスオーバーで代表業務を一時表示する
- 大分類ノードおよび展開中の代表業務群からポインターが外れたら閉じる

#### キーボード

- 大分類ノードへのフォーカスで代表業務を表示する
- Enterによる大分類の選択処理は持たない
- Tabで展開された代表業務へ移動できる

#### タッチ

- 大分類をタップすると代表業務を展開する
- これは大分類の選択ではなく、代表業務を表示するための操作とする

### 4.3 代表業務ノード

代表業務ノードは、全体地図から個別業務へ入る主な起点とする。

代表業務ノードのみクリック・Enterで選択できる。

代表業務を選択すると、当該業務を選択業務として確定し、対応する代表的な業務プロセスを表示する。

代表業務は、単に件数を均等配分するのではなく、各段階の意味を理解するために必要な業務を選定する。

代表業務の選定基準は次とする。

- 段階の開始または完了を表す
- 情報・状態・責任の引渡し点である
- 重要判断または例外分岐を含む
- 複数領域を横断する
- 初学者が段階の意味を理解しやすい

---

## 5. 業務プロセス

### 5.1 定義

業務ノードは一つの仕事を表す。

業務プロセスは、複数の業務ノードを、開始から終了まで意味のある順序でつないだ業務シナリオを表す。

```text
業務ノード
BM-09-06 点検異常を判定する

業務プロセス
点検計画から異常対応・報告まで
```

業務ノードと業務プロセスは一対一ではない。一つの業務が複数の業務プロセスへ含まれることを許容する。

### 5.2 表示範囲

選択業務の前後1ホップだけを表示しない。

選択業務が含まれる業務シナリオを、開始から終了まで表示する。

```text
開始
  ↓
点検を計画する
  ↓
点検を実施する
  ↓
点検値を記録する
  ↓
点検異常を判定する  ← 選択中
  ├─ 正常の場合
  │    ↓
  │  点検結果を承認する
  │    ↓
  │  点検結果を報告する
  │    ↓
  │  記録を保存する
  │
  └─ 異常の場合
       ↓
     不具合を受け付ける
       ↓
     緊急度を判断する
       ↓
     一次対応を行う
       ↓
     修繕・復旧の流れへ
```

### 5.3 方向

横方向を時間・進行方向とする。

通常の流れは左から右へ進める。

条件分岐は上下へ展開し、横方向の意味を維持する。

### 5.4 選択業務

選択業務をプロセス内で明確に強調する。

強調には、ラベル「選択中」と枠線・背景を併用し、色だけに依存しない。

### 5.5 複数プロセスへの所属

一つの業務が複数の業務プロセスへ含まれる場合、代表シナリオを初期表示する。

画面上部で、当該業務が含まれる別シナリオへ切り替えられるようにする。

```text
この業務が含まれる流れ
[点検異常から復旧まで ▼]
```

代表シナリオはデータ上で明示する。UIが接続数などから推測しない。

---

## 6. 線と経路の表現

### 6.1 基本原則

線は1種類だけとする。

```text
矢印 ＝ 次に進む
```

利用者へ線の色、太さ、破線、矢印形状の意味を覚えさせない。

### 6.2 条件分岐

条件は線種ではなく、日本語ラベルで表す。

```text
異常を判定する
  ├─ 正常の場合 → 結果を承認する
  └─ 異常の場合 → 不具合を受け付ける
```

### 6.3 別プロセスへの引渡し

点線などでは表現せず、行き先ノードとして表示する。

```text
一次対応を行う
  ↓
[修繕・復旧の流れへ]
```

行き先ノードを選択すると、対象業務プロセスへ切り替える。

### 6.4 差戻し・再実施

戻り線を画面内へ長く引かない。

```text
結果を確認する
  ├─ 問題なし → 報告する
  └─ 不備あり → [点検の再実施へ]
```

「○○へ戻る」ノードを選択すると、同一プロセス内の対象業務へフォーカスを移す。

### 6.5 合流

複数経路が同じ業務へ進む場合は、同じ業務ノードへ矢印を接続する。

合流専用の線種や記号は設けない。

---

## 7. 業務の詳細

選択業務の詳細は、業務プロセスとは別の右ペインに表示する。

表示項目は次とする。

- 業務ID・業務名
- 概要
- 開始契機
- 完了条件
- 入力
- 成果物
- 実施者
- 判断者・承認者
- 法令・基準
- 作業手順
- チェックリスト・帳票
- 関連業務
- 参照元

中央の業務プロセスへ、帳票、担当者、法令、類似業務などをノードとして混在させない。

中央は業務の進行、右ペインは業務に付随する情報という役割を維持する。

---

## 8. 検索

### 8.1 位置づけ

検索はすべての画面状態から利用できる共通機能とする。

検索を独立した第4画面にはしない。

### 8.2 検索対象

- 業務ID
- 業務名
- 業務説明
- 18業務領域
- ライフサイクル段階
- 業務プロセス名
- 入力
- 成果物
- 担当者
- 法令・基準
- 作業手順
- チェックリスト・帳票

### 8.3 検索結果

検索結果を種類ごとに分ける。

- 業務
- 業務プロセス
- 関連情報

業務を選択した場合は、その業務を選択業務として確定し、代表シナリオと業務詳細を表示する。

業務プロセスを選択した場合は、当該プロセスを開始から終了まで表示する。選択業務が未指定の場合は、プロセスの開始業務または代表業務を選択状態とする。

関連情報を選択した場合は、その情報に関係する業務を検索結果内で提示し、業務選択へつなげる。

## 10. URLと履歴

URLは、画面状態を再現できる情報を持つ。

### 全体地図

```text
/explorer/
```

必要に応じて、大分類プレビュー状態はURLへ保持しない。

### 業務プロセス

```text
/explorer/?business=BM-09-06&process=inspection-abnormality
```

保持する情報は次とする。

- 選択業務ID
- 表示中の業務プロセスID

ホバー・フォーカスによる代表業務プレビューは、URLとブラウザ履歴へ追加しない。

業務選択、業務プロセス切替、全体地図への復帰はブラウザ履歴へ追加する。

---

## 11. 業務プロセス正本データ

業務プロセスは、既存の`precedes`関係を無制限探索して生成しない。

各プロセスについて、開始、終了、順序、条件分岐、別プロセスへの引渡し、再実施を正本として定義する。

概念構造は次とする。

```yaml
id: inspection-abnormality
name: 点検計画から異常対応・報告まで
summary: 点検の計画から結果保存または異常対応への引渡しまでを示す
representative: true
entryBusinessIds:
  - BM-09-01
steps:
  - id: plan
    businessId: BM-09-01
    next: execute
  - id: execute
    businessId: BM-09-04
    next: record
  - id: record
    businessId: BM-09-05
    next: judge
  - id: judge
    businessId: BM-09-06
    branches:
      - label: 正常の場合
        next: approve
      - label: 異常の場合
        next: incident-handoff
  - id: approve
    businessId: BM-09-10
    next: complete
  - id: incident-handoff
    handoffProcessId: restoration
    label: 修繕・復旧の流れへ
  - id: complete
    terminal: true
```

必要なデータ要素は次とする。

- プロセスID
- プロセス名
- 初学者向け説明
- 代表プロセスかどうか
- 開始点
- 終了点
- 業務ノード
- 次工程
- 条件分岐ラベル
- 別プロセスへの引渡し
- 同一プロセス内の再実施先
- 当該プロセスへ含まれる業務ID一覧

## 12. 既存実装の扱い

| 現在の機能 | 方針 |
|---|---|
| 業務検索 | 検索対象を拡張して再利用する |
| 業務詳細パネル | 情報構造を整理して再利用する |
| URL・履歴同期 | `business`と`process`を中心に再利用する |
| React Flow／ELK | 全体地図・業務プロセス描画に再利用する |
| 1ホップの仕事の流れ | 廃止する |
| 全体から見る | 全体の業務地図へ置き換える |
| 関係から探す | 業務詳細と検索へ統合する |
| 下部ライフサイクル | 全体の業務地図へ統合する |
| 3表示タブ | 廃止する |

## 13. 明示的に採用しない設計

次の設計へ戻さない。

- 大分類ノードをクリックして選択する
- 全体地図と個別業務プロセスを同時に表示する
- 選択業務の前後1ホップだけを業務プロセスと呼ぶ
- 帳票、担当者、法令を中央の業務プロセス図へ混ぜる
- 線の色、太さ、破線で意味を区別する
- ホバーだけで選択業務やURLを変更する
- 接続関係を無制限探索してエンドツーエンドプロセスを自動推測する
- 4機能を同列タブとして並べる

---

## 14. 未確定事項

実装前に次を確定する必要がある。

1. 全体地図の大分類ノード配置
2. 各大分類の代表業務数と選定結果
3. 初期作成する業務プロセスの範囲と優先順位
4. 業務プロセス正本の保存形式
5. 分岐が多いプロセスの段階表示・折りたたみ方法
6. 全体地図へ戻った際に、直前の大分類プレビューを復元するか
7. タッチ端末で代表業務群を閉じる操作
8. 検索結果に関連情報をどの粒度まで含めるか

未確定事項は推測で実装せず、個別に合意してから確定する。

---

## 15. 受け入れ基準

- 初期画面を見て、ビルメンテナンス業務全体の地図であると理解できる
- 大分類ノードが業務選択対象ではないことが分かる
- 大分類へのホバー・フォーカス・タップで代表業務を確認できる
- 代表業務のクリック・Enterから業務プロセスへ移動できる
- 全体地図と業務プロセスが同時表示されない
- 選択業務が含まれる流れを開始から終了まで確認できる
- 一つの業務が含まれる複数プロセスを切り替えられる
- 矢印の意味を「次に進む」だけで理解できる
- 条件分岐、別プロセスへの引渡し、再実施を日本語で理解できる
- 業務詳細で入力、成果物、担当者、法令、手順を確認できる
- すべての状態から検索を利用できる
- マウス、キーボード、タッチで同等の主要操作を完了できる
- URL共有・再読み込み・戻る・進むで同じ状態を復元できる
