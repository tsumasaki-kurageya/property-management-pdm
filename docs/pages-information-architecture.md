# GitHub Pages 情報設計・編集方針

## 1. 目的

本書は、ビルメンテナンス業務の分析成果物を、ビルメンテナンス業務に詳しくないソフトウェア開発チームが順番に学べるドキュメントサイトへ再編集するための基準を定める。

サイトの目的は、製品やソフトウェアによる解決策を検討することではなく、ビルメンテナンス業務について一般的な全体像、関係者、仕事の流れ及び条件による違いを理解することである。

## 2. 対象読者と読後目標

### 2.1 主な対象読者

- ビルメンテナンス業務を担当した経験がないソフトウェア開発者
- 業界固有の略語、契約関係及び法定業務に馴染みがないチームメンバー
- 業務カタログを直接読む前に、業務の背景と全体像を理解したい人

読者はソフトウェア開発の知識を持つ場合があるが、本文ではその知識を前提にしない。ソフトウェア設計の用語への置き換えや、製品機能を想起させる説明は行わない。

### 2.2 読後に説明できる状態

本文を一通り読んだ人が、少なくとも次を自分の言葉で説明できる状態を目指す。

1. ビルメンテナンスが、建物を安全、衛生的かつ継続的に利用できる状態へ保つ仕事であること
2. 清掃、衛生、設備、警備・防災だけでなく、契約、計画、報告、請求及び改善まで含むこと
3. ビルオーナー、PM、FM、BM、専門業者、テナント等の関係と、実施者と意思決定者が一致しない場合があること
4. 契約から立ち上げ、計画、実施、報告、請求、改善へ至る基本的な循環
5. 定常業務から異常、修繕、是正及び緊急対応へ分岐する流れ
6. 作業実施、記録、確認、検収、報告、請求及び利用再開が別の完了状態であること
7. 建物用途、管理方式、契約役割、法令及び責任主体によって業務が変わること
8. 詳細を確認するとき、業務カタログ、プロセスマップ及び各プロファイルのどこを参照すればよいか

## 3. 情報構造の基本方針

サイトは、役割の異なる次の二層へ分ける。

| 層 | 目的 | 読み方 | 内容 |
|---|---|---|---|
| 学ぶための本文 | 初学者が全体像から順番に理解する | 推奨順序に沿って読む | 平易な説明、代表例、要点、図解 |
| 調べるためのリファレンス | 特定の業務や条件を確認する | 検索、索引、関連リンクから参照する | 181業務、12横断プロセス、各種プロファイル、手順・対応表 |

本文はリファレンスの短縮版ではない。読者が理解しやすい順序で複数の原本を横断して説明する。リファレンスは網羅性と追跡性を維持し、本文で省略した詳細を確認できるようにする。

## 4. 学習順序

本文は次の問いが順番に解ける構成とする。

| 段階 | 読者の問い | 主な理解内容 |
|---|---|---|
| 1. 入口 | 何をする業界なのか | 目的、対象範囲、建物利用との関係 |
| 2. 登場人物 | 誰が関わるのか | 関係者、契約関係、実施・判断・承認の違い |
| 3. 全体の流れ | 仕事はどう循環するのか | 契約から改善までの横断プロセス |
| 4. 日常の現場 | 普段は何をしているのか | 清掃、衛生、設備、警備・防災、共通動作 |
| 5. 周辺業務 | 現場作業を何が支えるのか | 計画、人員、資材、記録、報告、台帳、請求 |
| 6. 例外 | 異常時はどう動くのか | 受付、緊急度判断、安全確保、修繕、復旧、是正 |
| 7. 条件差 | なぜ物件ごとに違うのか | 用途、管理方式、契約役割、責任主体、法令 |
| 8. 詳細確認 | 個別業務をどこで調べるのか | カタログ、プロセス、プロファイル、用語集の使い方 |

法令や契約上の細部から説明を始めない。まず建物利用と日常業務のイメージを形成し、その後に責任や条件差を重ねる。

## 7. 原本とサイト本文の同期方針

### 7.1 正本

既存の分析成果物を正本として維持し、削除又はサイト本文へ置換しない。サイト本文は正本を初学者向けに再構成した派生文書とする。

### 7.2 変更順序

事実、業務ID、業務数、プロセス、責任分界又は条件差を変更するときは、次の順序で更新する。

1. 該当する分析用原本を更新する
2. 影響する本文ページと図を更新する
3. リファレンスの表示、リンク及び索引を確認する

本文だけで新しい業務事実や判断基準を追加しない。初学者向けの例示を追加する場合も、どの原本に基づくかを確認する。

### 7.3 ページの追跡情報

各本文ページは、Astro Starlightへ移行するときにfrontmatter等で次を保持する。

- 主な元資料
- 最終確認日
- 記載状態（確認済み、標準仮説、物件・契約依存）
- 関連する業務ID又はプロセスID

元資料へのリンクはページ末尾の「さらに詳しく」で提示し、本文中の読書を過度に中断させない。

## 8. 執筆ルール

### 8.1 文体と説明順

- 専門用語より先に、対象となる建物の状態や行為を平易に説明する
- 一文を短くし、一つの段落では一つの論点を扱う
- 原則として「目的・背景 → 典型的な流れ → 判断や例外 → 条件による違い → 詳細資料」の順に説明する
- 抽象説明だけで終わらせず、オフィスビル等の短い代表例を添える
- 代表例を業界全体の必須手順として断定しない
- 実施者、判断者、承認者、法的義務主体を混同しない
- 「作業完了」「技術的復旧」「検収」「利用再開」等、異なる状態をまとめて「完了」と表現しない

### 8.2 用語

- 初出時は日本語の正式名称を記載し、その後に略語を併記する
- 略語だけを見出しにしない。例：`プロパティマネジメント（PM）`
- 同じ概念には同じ語を使い、原本と異なる独自の言い換えを増やさない
- 一般語と業界用語で意味が異なる場合は、その場で短く説明し用語集へリンクする
- 類似語は差を説明する。例：点検と保守、修繕と工事、承認と検収、警備と防災

### 8.3 注記の区分

本文では次の区分を視覚的にも区別する。

| 区分 | 使用条件 | 表記例 |
|---|---|---|
| 要点 | 章の理解に不可欠な結論 | `要点` |
| 例 | 抽象概念を具体化する代表例 | `例` |
| 条件差 | 建物、契約、管理方式等で変わる事項 | `条件による違い` |
| 標準仮説 | 公的資料又は一般的構造から整理した未検証内容 | `標準モデル上の仮説` |
| 注意 | 安全、法令、権限又は適用上の誤解を防ぐ事項 | `注意` |

「通常」「一般的」「原則」と記載するだけで条件差を隠さない。現場ヒアリング未実施の内容は確定事実のように表現しない。

### 8.4 根拠と出典

- 法令、資格、法定周期、行政報告等は、公的機関又は法令本文を優先する
- 業界の説明は、業界団体、公的仕様書等の一次性が高い資料を優先する
- 出典は主張の近く又はページ末尾へ示し、出典一覧にも集約する
- 引用は必要最小限とし、本文は原則として要約する
- 法令情報には、建物条件、自治体、改正等により適用が変わる旨を明記する

## 9. 図解ルール

図は文章を飾るためではなく、関係、順序、分岐又は比較を理解しやすくするために使う。

| 伝えたい内容 | 基本形式 | 使用例 |
|---|---|---|
| 時間順の進行 | フロー図 | 契約から改善、異常から復旧 |
| 状態の変化 | 状態図又は段階図 | 実施から検収・利用再開 |
| 関係者と責任 | 責任関係図又は責任分界表 | オーナー、PM、FM、BM |
| 条件ごとの差 | 比較表 | 常駐・巡回・遠隔監視、用途差 |
| 分類・包含関係 | 概念図 | 業務領域、衛生管理の対象 |
| 詳細な対応関係 | 表 | 業務IDと手順ID、法令義務 |

### 9.1 制作基準

- 更新頻度が高い業務フロー、状態図及び関係図は、原則としてMarkdown内のMermaidで管理する
- 厳密な比較、ID対応、責任割当は図だけにせず表も用意する
- 図中の要素数を抑え、全体図と詳細図を分ける
- 色だけで区別せず、ラベル、線種又は形状を併用する
- 各図へタイトル、本文による要約及び代替テキスト相当の説明を付ける
- 図だけに新しい事実を記載せず、本文又は表からも同じ内容を確認できるようにする
- 写真やイラストを用いる場合は、特定現場の標準状態を保証するものとして使わない

## 10. 本文とリファレンスの境界

### 10.1 本文へ含めるもの

- 初学者が全体像を理解するために必要な代表的な業務と流れ
- 間違えやすい概念の違い
- 業務が変わる主な条件
- 代表例と、例外があることの説明
- 詳細な原本へ進むための導線

### 10.2 リファレンスへ置くもの

- 181業務の全項目
- 12横断プロセスの詳細な業務ID列
- 14重要業務の全分析項目
- 用途別、管理方式別、契約役割別、法令別の詳細マトリクス
- 個別の現場作業手順、チェックリスト及び対応表
- 出典URLの網羅的な一覧

### 10.3 扱わないもの

- ソフトウェア製品の機能、画面、データモデル又はアーキテクチャ
- 製品による解決策、製品カバレッジ及び競合比較
- 将来機能のアイデア又は要件候補
- 特定の会社、顧客又は物件だけに適用される運用
- 個別物件へそのまま適用できると誤解される作業指示
- 法律上の助言又は個別施設への法令適用の断定

業務上の非効率や問題仮説は、ビルメンテナンス業務を理解するために必要な範囲では記載できる。ただし、ソフトウェアでの解決方法へ接続しない。

## 11. ページ共通構成

本文ページは原則として次の構成を使う。内容に合わない見出しを機械的に残す必要はない。

1. このページで分かること
2. 業務の目的又は背景
3. 全体像
4. 典型的な流れ又は構成要素
5. 主な判断、例外又は間違えやすい点
6. 建物・契約等による違い
7. まとめ
8. 関連ページ
9. さらに詳しく（分析用原本・出典）

各ページ冒頭で前提知識を明示し、本文中に未説明の略語を持ち込まない。長いページは一つの問いを単位に分割し、同じ説明を複数ページへ複製しない。

## 12. ナビゲーションと検索

- サイドバーでは「はじめに」「業務の全体像」「現場の業務」「異常と周辺業務」「条件による違い」「リファレンス」を第一階層とする
- 本文ページには前後の学習ページへの導線を置く
- 用語、業務ID及びプロセスIDから検索できるようにする
- 本文中の業務IDは該当する業務カタログ項目へリンクする
- プロセスから関連業務へ、業務から関連プロセス・手順・条件差へ移動できるようにする
- 初学者向け本文と分析用原本を、同じサイドバー階層へ無秩序に混在させない

## 13. 初版の完成条件

情報設計に従った初版は、次を満たしたとき完成とする。

- 学ぶための本文と調べるためのリファレンスがナビゲーション上で区別されている
- 本文を推奨順序で読むと、2.2の読後目標を一通り満たせる
- 各本文ページから主な分析用原本へ到達できる
- 原本の業務ID、プロセスID、件数及び用語と本文に矛盾がない
- 標準仮説、条件依存及び法令上の注意が区別されている
- 主要な関係、流れ、分岐及び比較に図又は表がある
- 図を見なくても本文から要点を理解できる
- 内部リンク切れ、表示崩れ及び検索不能な主要用語がない
- 製品機能、製品カバレッジ又は解決策の検討が混入していない
